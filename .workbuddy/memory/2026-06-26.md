## 设置页可转发 (17:30)

为 5 个设置相关页面（profile / member-center / member / staff-settings / activity）加微信原生分享（好友/群 + 朋友圈）。

### 改动总览

每个页面都需要 3 处改动：
1. `.js` — `onShow` 中调 `wx.showShareMenu({ menus: ['shareAppMessage', 'shareTimeline'] })`
2. `.js` — `Page({})` 内加 `onShareAppMessage()` 和 `onShareTimeline()` 返回 `{ title, path }` / `{ title, query }`
3. `.wxml` — 加 `<button open-type="share">` 分享按钮（button 标签不可替换为 view，open-type="share" 必须用 button）

### 关键 bug 修复
`pages/activity/activity.js` 第 104 行的 `activityId` 取值链原本是：
```js
options.id || options.activity_id || scene.a || scene.activity_id
```
分享进入时 `?a=xxx` 出现在 `options.a` 而非 `options.scene` 中（scene 仅 wxacode 扫码进入才存在），导致分享活动页打不开。
修复为：
```js
options.id || options.a || options.activity_id || scene.a || scene.activity_id
```

### 设计原则
- 不配 `app.json` 全局 `enableShareAppMessage`/`enableShareTimeline` —— 避免激活其他 13 个非设置页

### 踩坑
- `open-type="share"` 必须是 `<button>`，不能 `<view bindtap>`

- button 默认样式要 reset：`::after { border: 0; }` + 重置 background/padding
- 列表行用 button 时需重置为左对齐、`width:100%`、无圆角

## reservation-detail 加分享 (18:12)

第一轮决定"不做"——担心联系人/手机号/订单号泄露。这次重做时采用**脱敏拼接**方案：

### 分享文案（动态）
- 格式：`我在停云山居预订{山间的一餐/山居的一宿}，{房间名} · {状态}`
- 住宿 → "山居的一宿"；用餐 → "山间的一餐"
- 房名从 `order.room_snapshots[].name` 读（不取整段 room_names 列表，控制长度）
- 状态取 `order.status_text`（待支付/已确认/已完成/已取消...）
- **不含**：联系人姓名、手机号、订单号、用户备注、金额
- 标题硬切 ≤ 30 字
- path 指向 home（不指回自己，朋友收到也打不开这个订单）

### 分享图
- 从 `order.room_snapshots[0].image_url` 读第一间房/包间图，作为 imageUrl
- 这是云存储临时链接，理论上 1-2 小时后会过期——分享图若变灰是预期行为

### onShareTimeline 的 query
- 朋友圈支持 query 参数，会拼到 path 后
- 这里挂 `id={order_no}`，分享到朋友圈的用户从朋友圈点进来会带着 `?id=xxx` 进入 home——home 不读这个 id，会被丢弃
- 实际效果：朋友圈用户点开进 home，不进订单详情
- 以后如要做"朋友圈看到某订单"，需要在 home onLoad 里识别这个 id 并跳转

### 为什么不做"分享时跳自己"
- 朋友的 openid / 订单归属和当前用户不匹配
- 即使朋友点了进 reservation-detail，onLoad 时按 order_no 查云函数查不到，会跳回 orders 页
- path 永远指 home 是最安全的兜底
