# 云开发托管、微信支付与打印机接入说明

整理日期：2026-06-10

适用项目：停云山居小程序、后台管理页、扫码点餐与订单打印

## 一句话结论

当前后台页面继续用静态网站托管即可；打印机和微信支付不能只靠静态网站托管完成，必须放到云函数或云托管这类服务端环境里处理。

对于本项目，建议先用现有云函数接入芯烨云打印；微信支付如果想省开发和配置成本，优先考虑云开发 199 标准版的支付集成能力。如果买 19.9 个人版再加云托管，也可以自己开发微信支付后端，但不会自动获得 199 标准版的内置支付集成能力。

## 静态网站托管和云托管的区别

| 对比项 | 静态网站托管 | 云托管 / Express 服务 |
| --- | --- | --- |
| 本质 | 存放 HTML、CSS、JS、图片等静态资源 | 运行 Node.js、Express 等后端服务 |
| 是否能运行后端代码 | 不能 | 可以 |
| 是否适合放后台管理页 | 适合 | 可以但通常没必要 |
| 是否适合放敏感密钥 | 不适合 | 适合放在服务端环境变量中 |
| 是否适合调用打印机开放接口 | 不适合直接调用 | 适合 |
| 是否适合处理微信支付回调 | 不适合 | 适合 |
| 成本特征 | 主要是存储、访问流量 | CPU、内存、实例、构建、公网流量等 |
| 项目中的定位 | 管理后台页面、H5 页面 | 统一后端 API、支付回调、第三方接口服务 |

本项目里：

- `tysj8` 是 Vite 静态站，配置在 `tysj8/cloudbaserc.json`。
- `admin-web` 是后台管理静态页面。
- 小程序业务逻辑主要在 `tingyun2026/cloudfunctions` 里的云函数中。
- 截图里的 `express-vr9p` 是云托管 Express 服务，不等同于静态托管。

## 当前项目建议架构

```text
小程序 / 后台静态网页
  -> 调用云函数或云托管后端
    -> 创建订单、预约、报名等业务数据
    -> 触发微信支付
    -> 支付成功后处理回调
    -> 调用芯烨云打印接口
    -> 保存打印状态、支付状态、订单状态
```

静态网站托管只负责展示页面。凡是涉及密钥、签名、支付、打印、回调、数据库写入校验的逻辑，都应该放到云函数或云托管后端。

## 打印机接入是否需要云托管

短期不需要专门为了打印机上云托管。

芯烨云打印接口需要开发者 ID、`UserKEY`、打印机 SN，并且需要服务端生成签名。如果这些信息放在小程序端或静态网页里，会有泄露风险。

推荐方案：

```text
用户扫码点餐 / 提交预约
  -> 小程序调用云函数
  -> 云函数写入订单
  -> 云函数生成小票内容
  -> 云函数调用芯烨云打印接口
  -> 保存打印订单号和打印状态
```

优先接入位置：

- `tingyun2026/cloudfunctions/mealOrderManage/index.js`
- `tingyun2026/cloudfunctions/reservationManage/index.js`

后续如果出现这些需求，再考虑云托管：

- 需要统一 REST API 服务。
- 需要接收芯烨云状态回调。
- 需要更完整的打印队列、失败重试、补打任务。
- 云函数调用第三方接口受限或维护不方便。
- 多个第三方服务都需要统一后端承接。

## 微信支付和云开发套餐

云开发套餐里的 19.9 和 199 不是简单的“流量大小区别”，它们在能力支持上也有差异。

根据官方说明，云开发标准版及以上支持微信支付集成。这里的“支持微信支付集成”，更偏向云开发内置的微信支付能力、模板或更低成本的接入方式。

因此：

- 买 199 标准版：更适合直接使用云开发官方支付集成能力。
- 买 19.9 个人版 + 云托管：可以买到一个可运行后端的环境，但不等于自动拥有 199 标准版的支付集成能力。
- 如果自己写微信支付 V3 后端，19.9 + 云函数或云托管理论上也可以实现，但需要自己处理商户号、APIv3 密钥、证书、下单、回调验签、订单查询、退款等细节。

## 19.9 + 云托管能不能做微信支付

可以分两种情况看。

### 方案 A：使用云开发内置微信支付集成

建议购买 199 标准版。

优点：

- 接入路径更接近官方云开发文档。
- 支付相关配置和云函数集成更省事。
- 更适合当前项目这种小程序订单支付场景。

缺点：

- 固定月费更高。

### 方案 B：19.9 + 自己写微信支付 V3 后端

理论上可以，但开发复杂度更高。

需要准备：

- 已认证的小程序。
- 微信支付商户号。
- 商户号和小程序 AppID 绑定。
- APIv3 密钥、商户证书或平台证书相关配置。
- 后端接口：统一下单、支付回调、订单查询、退款、重复通知处理。
- 小程序端调用 `wx.requestPayment`。

这个方案的关键点是：云托管只是提供后端运行环境，它不是“支付功能开关”。买云托管不会把 19.9 个人版升级成 199 标准版能力。

## 推荐采购和实施顺序

### 第一阶段：保持当前低成本结构

- 后台页面继续使用静态网站托管。
- 小程序继续调用云函数。
- 打印机接入先放在云函数里。
- 订单、预约、打印状态写入云数据库。

适合当前业务验证阶段。

### 第二阶段：接入真实微信支付

优先建议：

- 如果希望省开发时间，选择云开发 199 标准版并按官方支付集成方式做。
- 如果希望省固定成本，选择 19.9 并自行接微信支付 V3 API，但要接受更高开发成本。

### 第三阶段：视情况引入云托管

只有当云函数不够用，或者后端 API 复杂度明显上升时，再把部分服务迁到云托管。

典型适用场景：

- 统一 `/api/orders`、`/api/payments`、`/api/printer` 等后端接口。
- 处理第三方回调。
- 更复杂的队列、重试、日志和监控。
- 未来后台系统需要完整服务端渲染或独立后端。

## 当前项目的建议结论

当前不要为了打印机单独买云托管。

建议先这样走：

1. 静态网站托管继续放后台管理页。
2. 云函数接入芯烨云打印。
3. 支付上线前确认是否购买 199 标准版。
4. 如果选择 19.9 方案，就按微信支付 V3 自研后端方案评估开发量。
5. 云托管作为后续扩展项，不作为当前必须项。
