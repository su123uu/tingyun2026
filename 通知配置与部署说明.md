# 通知配置与部署说明

## 1. 通知渠道

- 微信小程序订阅消息：点餐状态、预订状态、活动报名、会员消费。
- 企业微信：店员新单、加菜、支付成功、待核销和后台状态变更。
- 厨房打印：点餐按提交批次打印；预订在支付成功后打印。

所有发送都由 `notificationManage` 或 `payCallback` 写入 `notification_logs`。订阅消息授权一次只能发送一次，不能复用。

## 2. 点餐事件规则

| 事件 | 微信客户 | 企业微信 | 打印 |
| --- | --- | --- | --- |
| 首单/加菜提交 | `mealOrderStatus`：制作中 | 新单/加菜 | 本次批次 |
| 后台更新为已完成 | 不发送（首单/加菜已使用本次订阅授权） | 点餐状态更新 | 无 |
| 散客支付回调成功 | 不重复发送 | 已结账 | 不重复打印 |
| 会员确认积分抵扣 | 无 | 待会员核销；订单已清台 | 无 |
| 后台会员核销完成 | `memberConsumption` | 核销完成 | 无 |

会员消费订阅授权在会员结账前请求，并以结账人的 `checkout_openid` 为接收人。

## 3. 环境变量

在云函数环境配置实际值，禁止写入代码库或操作文档：

```text
WX_SUBSCRIBE_MEAL_ORDER_STATUS_TEMPLATE_ID=
WX_SUBSCRIBE_RESERVATION_STATUS_TEMPLATE_ID=
WX_SUBSCRIBE_MEMBER_CONSUMPTION_TEMPLATE_ID=
WX_SUBSCRIBE_ACTIVITY_SIGNUP_SUCCESS_TEMPLATE_ID=
WX_SUBSCRIBE_DINING_RESERVATION_STATUS_TEMPLATE_ID=
WX_SUBSCRIBE_MINIPROGRAM_STATE=formal

WECOM_CORP_ID=
WECOM_AGENT_ID=
WECOM_APP_SECRET=
WECOM_TO_USER=
# 或 WECOM_WEBHOOK_URL=

WX_APPID=
WX_APP_SECRET=
```

模板字段 key 可用 `WX_SUBSCRIBE_*_FIELD` 覆盖默认配置。企业微信可选择应用消息或群机器人 Webhook，不要同时依赖两者。

订阅消息投递版本必须与当前测试阶段一致：在微信开发者工具或开发版验收时配置 `WX_SUBSCRIBE_MINIPROGRAM_STATE=developer`；上传体验版时使用 `trial`；小程序正式发布后再改回 `formal`。否则用户即使已允许订阅，微信接口也可能拒绝向当前版本投递。

## 4. 部署与验证

部署 `notificationManage`、`mealOrderManage`、`payCallback`、`reservationManage`、`activitySignupManage`、`adminManage`、`initDatabase` 后，依次验证：

1. 客户允许订阅后提交点餐，检查客户消息、企业微信、打印和日志。
2. 散客整桌支付成功，检查回调状态与企业微信已结账消息。
3. 会员整桌确认积分抵扣后，检查桌台已释放且订单进入 `offline_pending`；在后台完成核销后，检查会员消费模板和企业微信核销消息。
4. 预订与活动的支付、核销、确认消息均能在 `notification_logs` 定位结果。

排查点餐通知时，先按订单号检查两个集合：`notification_subscriptions` 有 `accepted` 记录表示用户本次授权已被写入；`notification_logs` 的 `sent` 表示微信已接受投递，`failed` 或 `skipped` 会记录具体原因。两处都没有记录通常表示 `notificationManage` 未部署到当前云环境，或当前运行的 `mealOrderManage` 仍是旧版本。
