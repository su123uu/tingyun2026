# 停云山居小程序需求文档

> 版本：运行版基线（2026-06-25）
> 本文描述当前小程序、云函数与后台应共同实现的行为；未实现能力会明确标注，不以早期原型为准。

## 1. 产品范围

停云山居小程序提供扫码点餐、用餐与住宿预订、活动报名、会员权益和订单查询。后台负责内容、菜品、桌台、房间、活动、会员和订单状态管理。

- 散客：授权手机号未匹配有效会员档案的用户。
- 会员：授权手机号匹配到有效会员档案的用户。
- 会员余额、积分及线下账户不在小程序展示或自动扣减；点餐会员可在订单详情确认使用积分抵扣，系统只记录抵扣金额并交由店员核销。

## 2. 扫码点餐

### 2.1 桌台与下单

1. 用户未扫码时可浏览菜单、加入购物车，但不能提交菜品。
2. 扫描有效桌码后，首次开台填写用餐人数并建立桌台会话。
3. 已开台且有活动订单的桌码再次被扫描时，直接进入该桌订单详情。
4. 提交菜品只创建首单或追加批次：每桌一次用餐只有一个 `meal_orders` 主订单，加菜追加至 `batches`。
5. 每次提交菜品立即通知厨房并打印本次批次；确认订单页不调起微信支付。

### 2.2 双价格与整桌结账

每个菜品快照保存普通价与会员价。订单详情展示整桌普通价、整桌会员价，以及首单/加菜各自的两种小计。

- 散客在订单详情结账：整桌按普通价发起一次微信支付。
- 会员在订单详情结账：整桌按会员价弹出“确认积分抵扣”窗口，确认后订单进入 `offline_pending`，由店员线下核销。
- 结账身份以实际点击结账并已完成手机号识别的用户为准，不以开台人或点菜人决定。
- 点餐页与确认订单页的会员身份以当前授权用户为准，桌台会话只记录桌号、人数与订单会话，不绑定开台人身份。
- 未配置会员价的菜品，会员价自动等于普通价。
- 散客微信支付成功后，订单自动完成并清台。
- 会员确认积分抵扣后，订单置为 `completed` + `offline_pending` 并立即清台；店员后续在后台完成核销后，支付状态改为 `settled`。
- 后台桌台总览展示菜品总价，并支持“编辑价格”：调整整桌总价、设置单个菜品免费。保存后更新订单及批次金额快照。

### 2.3 点餐状态

| 字段 | 值 | 含义 |
| --- | --- | --- |
| `order_status` | `preparing` / `completed` | 出餐主状态 |
| `payment_status` | `unpaid` | 已点菜、待整桌结账 |
|  | `paying` | 微信支付已发起，等待回调 |
|  | `offline_pending` | 会员待线下核销；点餐会员确认积分抵扣后，桌台已清但支付仍待核销 |
|  | `settled` | 微信支付成功或线下核销完成 |

## 3. 预订与活动

- 用餐预订：至少 6 人；散客支付成功后占用包间并待确认，会员提交后占用包间并待线下核销。
- 住宿预订：散客按普通房价、会员按会员房价；可使用已配置的免费住宿次数权益。
- 活动报名：公开或仅会员；收费散客微信支付，收费会员按会员价进入 `offline_pending`；免费活动允许用户取消。
- 预订和活动与点餐统一使用 `offline_pending` 表示会员待线下核销。

## 4. 会员与手机号

- 所有正式页面使用微信 `getPhoneNumber` 授权后匹配会员，不展示固定测试手机号或模拟绑定入口。
- 会员中心展示后台配置的服务权益、次数权益和活动记录。
- 小程序不展示会员余额、积分余额、积分明细或线下扣减记录。

## 5. 消息与打印

| 事件 | 客户微信订阅消息 | 企业微信店员消息 | 厨房打印 |
| --- | --- | --- | --- |
| 点餐首单/加菜提交 | 点餐状态：制作中 | 新单/加菜明细 | 本次批次 |
| 散客微信支付成功 | 不重复消耗已使用订阅授权 | 已结账 | 不重复打印 |
| 会员确认积分抵扣 | 无 | 待会员核销 | 不重复打印 |
| 后台完成会员核销 | 会员消费通知 | 核销完成 | 无 |
| 预订、活动状态变更 | 按已授权模板发送 | 状态变更通知 | 预订支付成功时打印 |

订阅消息为一次性授权；同一授权只用于对应事件。所有发送结果写入 `notification_logs`。

## 6. 当前不支持

- 会员余额、积分、次数的点餐自动扣减。
- 用户侧点餐退款与支付后取消。
- 点餐库存锁定。
- 旧版逐批支付订单兼容；测试数据可直接清空后启用当前流程。
