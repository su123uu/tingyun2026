# 停云山居小程序 · 业务逻辑设计 v1.0

> 制定日期：2026-06-01
> 依据：《停云山居小程序-需求文档.md》及业务讨论结果
> 用途：指导前端模拟逻辑、云数据库设计、云函数设计和后续后台开发

---

## 一、设计原则

1. 先完成业务设计，再逐步迁移 HTML 原型和开发微信小程序。
2. 当前阶段可使用模拟数据跑通流程，后续接入腾讯云开发、微信支付和后台。
3. 页面只负责展示和收集输入，金额、资格、房间占用和状态变化必须由业务服务校验。
4. 会员积分由现有线下积分程序管理。本期小程序不展示积分余额、积分明细和积分抵扣记录，只预留数据字段。
5. 用户端不提供预约退款按钮。需要取消预约或申请退款时，统一引导用户联系客服。

## 二、用户身份

### 2.1 用户类型

| 类型 | 判定方式 | 结算方式 |
|------|----------|----------|
| 散客 | 未授权手机号，或手机号未匹配会员档案 | 微信支付 |
| 会员 | 授权手机号后，匹配数据库中预先录入的有效会员档案 | 优先由店员在线下积分程序扣款；积分不足时补微信支付 |

### 2.2 会员绑定流程

```text
用户进入小程序
  → 请求微信手机号授权
  → 使用手机号查询会员档案
      ├─ 匹配到有效会员：绑定会员身份
      └─ 未匹配或拒绝授权：按散客处理
```

说明：

- 会员信息先由后台或数据库预先填写。
- 用户拒绝授权时仍可浏览页面，并按散客流程消费。
- 会员权益过期时，保留会员档案，但消费结算是否继续执行会员价需要由后台配置。本期默认仅有效会员享受会员价。

## 四、扫码点餐

### 4.1 基础流程

```text
扫描桌上二维码
  → 解析并校验桌号、二维码令牌和桌台启用状态
  → 填写用餐人数
  → 浏览菜单并加入购物车
  → 提交订单
  → 根据用户身份进入结算流程
  → 立即通知厨房
```

### 4.2 浏览与提交规则

- 从首页或底部导航进入点餐页时，允许浏览菜单和添加购物车。
- 只有扫描桌上二维码并成功获得桌号后，才允许提交订单。
- 桌号来自后台维护的 `meal_tables`，支持新增、停用、软删除和重新生成二维码。
- 正式云端二维码应包含 `table_id` 和 `qr_token`；桌台停用、删除或二维码重新生成后，旧二维码不可再创建用餐会话。
- 提交前必须填写用餐人数。
- 菜品金额在提交时重新计算，不能直接信任页面传入的金额。
- 首单提交后允许继续加菜。

### 4.3 散客点餐结算

```text
提交订单
  → 微信支付
  → 支付成功
  → 通知厨房
  → 制作中
  → 已完成
```

### 4.4 会员点餐结算

```text
提交订单
  → 立即通知厨房
  → 店员在线下积分程序扣款
      ├─ 积分足够：登记已扣积分
      └─ 积分不足：现场补微信支付
  → 制作中
  → 已完成
```

会员点餐不等待线下扣款完成再通知厨房，以免影响出餐速度。

### 4.5 点餐订单双状态

点餐订单同时维护厨房状态和结算状态。两个状态彼此独立。

厨房状态 `kitchen_status`：

```text
pending_notice → kitchen_notified → preparing → completed
```

| 状态 | 含义 |
|------|------|
| `pending_notice` | 已提交，正在通知厨房 |
| `kitchen_notified` | 厨房已收到订单 |
| `preparing` | 制作中 |
| `completed` | 已完成 |

结算状态 `settlement_status`：

```text
散客：pending_wechat_pay → settled

会员：pending_offline_points
        ├─ points_deducted → settled
        └─ pending_supplement → settled
```

| 状态 | 含义 |
|------|------|
| `pending_wechat_pay` | 散客待微信支付 |
| `pending_offline_points` | 会员待线下积分扣款 |
| `points_deducted` | 已在线下积分程序扣款 |
| `pending_supplement` | 积分不足，待补差额 |
| `settled` | 已结清 |

### 4.6 追加菜品

- 每次加菜生成一笔独立子订单，并关联桌台主订单。
- 当前 mock 使用 `table_session.session_id` 判断同一批客人：同一桌台会话内首次下单为首单，后续下单为追加菜子订单。
- 子订单单独通知厨房，避免漏单。
- 订单列表只显示一条主订单。
- 订单详情展示首单和历次加菜记录，并汇总总金额。
- 每笔首单或子订单分别记录结算状态，便于店员核对。

## 六、用餐预订

### 6.1 表单内容

| 字段 | 说明 |
|------|------|
| 用餐日期 | 不早于当天 |
| 用餐时段 | 午餐 `11:00-14:00`、晚餐 `17:00-21:00` |
| 用餐人数 | 至少 6 人 |
| 包间 | 根据日期、时段和人数筛选，可按规则多选 |
| 餐标 | 团餐、养云、归云、停云 |
| 联系人 | 必填 |
| 手机号 | 必填 |
| 备注 | 可选 |

### 6.2 餐标

| 餐标 ID | 名称 | 单价 |
|---------|------|-----:|
| `group_meal` | 团餐 | 40 元 / 人 |
| `yangyun` | 养云 | 168 元 / 人 |
| `guiyun` | 归云 | 218 元 / 人 |
| `tingyun` | 停云 | 318 元 / 人 |

交互要求：

- 餐标以卡片形式展示。
- 用户点击餐标图标后，可查看餐标详情。
- 餐标详情内容由后台配置。

### 6.3 金额计算

```text
用餐预订总价 = 用餐人数 × 餐标单价
```

当前不额外收取包间费。如未来新增服务费或包间费，应独立展示。

### 6.4 包间筛选与选择

现有包间：

| 包间 ID | 名称 | 建议容量 |
|---------|------|----------|
| `xigu` | 兮古 | 6-10 人 |
| `qilu` | 栖鹿 | 8-12 人 |
| `mengtang` | 梦堂 | 8-12 人 |
| `xinyu` | 心语 | 8-12 人 |

规则：

1. 用户必须填写日期、时段和人数后，页面才展示可用包间。
2. 已被占用的包间不可选择。
3. 少于 6 人禁止提交预订。
4. `6-12` 人最多选择 1 个包间；单个包间最大容量不足时，显示“人数不合适”并禁止选择。
5. `13-22` 人最多选择 2 个包间；`23` 人及以上可选择全部包间。
6. 所选包间总容量必须覆盖用餐人数。
7. 用户至少选择 1 个包间。

### 6.5 散客流程

```text
选择日期、时段、人数、可用包间和餐标
  → 计算总价
  → 最终校验并锁定包间 15 分钟
  → 微信支付
  → 已支付，待店长确认
      ├─ 店长确认：已确认
      └─ 店长拒绝：退款中 → 已退款
```

### 6.6 会员流程

```text
选择日期、时段、人数、可用包间和餐标
  → 计算总价
  → 提交预约并占用包间
  → 待确认、待线下积分扣款
  → 客服联系会员
      ├─ 积分足够：线下积分程序扣款
      └─ 积分不足：补微信支付
  → 店长确认：已确认
```

## 七、住宿预订

### 7.1 表单内容

| 字段 | 说明 |
|------|------|
| 入住日期 | 不早于当天 |
| 离店日期 | 必须晚于入住日期 |
| 入住晚数 | 自动计算，至少 1 晚 |
| 入住人数 | 至少 1 人，仅作为接待信息记录 |
| 房间 | 根据日期筛选，可多选 |
| 联系人 | 必填 |
| 手机号 | 必填 |
| 备注 | 可选，可填写儿童入住等需求 |

### 7.2 房间筛选

- 用户先选择入住日期和离店日期。
- 页面只展示当前日期范围内可用的房间。
- 所选房间在整个入住日期范围内均不可与其他占用记录冲突。
- 住宿预订不按房间容量限制提交，人数用于客服接待确认。
- 允许儿童入住，不提供加床。
- 会员进入住宿预订时，需要读取其可用住宿权益；若存在 `free_accommodation` 免费住宿权益，页面展示可用次数。
- 免费住宿权益不限房型，但仍需要按入住日期检查房间是否可用，并保留人工确认环节。

### 7.3 金额计算

散客：

```text
住宿总价 = 所选房间原价之和 × 入住晚数
```

会员：

```text
住宿总价 = 所选房间会员价之和 × 入住晚数
```

会员使用免费住宿权益：

```text
住宿应付金额 = 住宿总价 - 免费住宿权益抵扣金额
```

说明：

- 免费住宿权益从 `member_benefit_accounts` 读取，权益键为 `free_accommodation`。
- 用户提交住宿预订时，如果选择使用免费住宿权益，先锁定对应权益次数。
- 店长确认并完成服务后正式核销；若拒绝、退款或客服取消，则释放已锁定权益。

### 7.4 散客流程

```text
选择日期、人数和可用房间
  → 按原价计算总价
  → 最终校验并锁定房间 15 分钟
  → 微信支付
  → 已支付，待店长确认
      ├─ 店长确认：已确认
      └─ 店长拒绝：退款中 → 已退款
```

### 7.5 会员流程

```text
选择日期、人数和可用房间
  → 读取可用免费住宿权益
  → 按会员价计算总价，可选择使用免费住宿权益
  → 提交预约并占用房间
  → 待确认、待线下积分扣款或待权益核销
  → 客服联系会员
      ├─ 使用免费住宿权益：后台确认后核销权益
      ├─ 积分足够：线下积分程序扣款
      └─ 积分不足：补微信支付
  → 店长确认：已确认
```

## 八、预约状态与占用

### 8.1 预约业务状态

预约状态 `reservation_status`：

```text
pending_payment
  → paid_pending_confirmation
      ├─ confirmed → completed
      └─ refunding → refunded

pending_confirmation
  ├─ confirmed → completed
  └─ rejected
```

| 状态 | 适用场景 | 含义 |
|------|----------|------|
| `pending_payment` | 散客 | 已创建临时锁定，待微信支付 |
| `paid_pending_confirmation` | 散客 | 已支付，待店长确认 |
| `pending_confirmation` | 会员 | 已提交，待客服和店长确认 |
| `confirmed` | 全部 | 已确认接待 |
| `rejected` | 会员 | 无法接待，已拒绝 |
| `refunding` | 散客 | 无法接待，客服处理中 |
| `refunded` | 散客 | 微信退款完成 |
| `completed` | 全部 | 已完成服务 |
| `cancelled` | 全部 | 客服已取消 |

### 8.2 预约结算状态

预约结算状态 `settlement_status`：

| 状态 | 含义 |
|------|------|
| `pending_wechat_pay` | 散客待微信支付 |
| `wechat_paid` | 散客已微信支付 |
| `pending_offline_points` | 会员待线下积分扣款 |
| `points_deducted` | 会员积分已扣除 |
| `pending_supplement` | 积分不足，待补差额 |
| `settled` | 已结清 |
| `refunding` | 微信退款处理中 |
| `refunded` | 微信退款完成 |

### 8.3 房间占用规则

占用房间或包间的状态：

- 散客付款前的临时锁定，最长 `15 分钟`。
- `paid_pending_confirmation`。
- `pending_confirmation`。
- `confirmed`。

释放房间或包间的状态：

- 临时锁定超过 `15 分钟`未支付。
- `rejected`。
- `refunded`。
- `cancelled`。

店长确认时必须再次检查冲突。

### 8.4 为什么仍需人工确认

即使页面已筛选可用房间，仍保留人工确认：

- 房间可能临时维修、清洁未完成或被线下渠道预留。
- 用户可能填写儿童入住、聚餐布置或其他特殊需求。
- 会员需要核对线下积分扣款。
- 发生特殊情况时，客服可以及时联系用户并处理退款。

### 8.5 用户端操作

- 用户端不展示取消预约或退款按钮。
- 订单详情统一展示：“如需取消预订，请联系客服。”
- 退款由客服在后台发起。

每场活动支持配置：

| 字段 | 说明 |
|------|------|
| 活动标题 | 必填 |
| 活动详情 | 图文内容 |
| 活动时间 | 必填 |
| 活动地点 | 必填 |

| 是否收费 | 免费或收费 |

### 9.2 报名资格

- 每场活动由后台控制是否仅限会员。
- 公开活动允许散客和会员报名。
- 仅限会员活动要求手机号已绑定有效会员档案。
- 每位用户默认最多报名 2 人；超过 2 人需要联系客服。
- 报名人数不能超过剩余名额。

### 9.3 免费活动

```text
提交报名
  → 待确认
      ├─ 店员确认：已确认
      ├─ 店员拒绝：未通过
      └─ 用户自行取消：已取消并释放名额
```

免费活动允许用户自行取消。

### 9.4 收费活动

散客：

```text
提交报名
  → 按散客价微信支付
  → 已支付，待确认
      ├─ 店员确认：已确认
      └─ 无法参加：联系客服处理退款
```

会员：

```text
提交报名
  → 按会员价待线下积分扣款
  → 待确认
      ├─ 积分足够：线下积分程序扣款
      └─ 积分不足：补微信支付
  → 店员确认：已确认
```

收费活动取消或退款统一联系客服，不提供自助退款按钮。

## 十一、核心数据对象

### 11.1 用户 `users`

| 字段 | 类型 | 说明 |

| `user_id` | string | 用户 ID |
| `openid` | string | 微信用户标识 |
| `mobile` | string | 授权手机号 |

### 11.2 会员档案 `members`

| 字段 | 类型 | 说明 |
|------|------|------|

| `mobile` | string | 用于匹配授权手机号 |

## 十二、后台业务能力

本期先设计逻辑和数据结构，不制作后台页面。

### 12.1 预约管理

- 查看待确认的用餐和住宿预约。
- 查看房间与包间占用日历。
- 确认预约。
- 拒绝预约并填写原因。
- 为散客发起微信退款。
- 手动取消预约并释放占用。
- 登记会员线下积分已扣款。
- 登记积分不足和待补差额。

### 12.2 点餐管理

- 新增、修改、停用和删除桌台。
- 为单个桌台生成或重新生成二维码，下载后打印。
- 查看桌台首单和追加订单。
- 查看待线下扣款的会员订单。
- 登记已扣积分或已补差额。
- 推进厨房状态：已通知、制作中、已完成。
- 查看厨房通知或云打印异常。

### 12.3 活动管理

- 创建和编辑活动。
- 设置报名范围、收费方式、散客价和会员价。
- 设置人数上限和报名截止时间。
- 确认或拒绝报名。
- 登记会员线下积分扣款。
- 为散客收费报名处理微信退款。

## 十四、推荐开发顺序

### 第一阶段：基础能力

1. 手机号授权与会员身份模拟。
2. 统一模拟数据和本地存储。
3. 金额计算、手机号校验、日期校验等公共方法。

### 第二阶段：点餐闭环

1. 扫码建立桌台会话。
2. 购物车。
3. 散客微信支付模拟。
4. 会员先通知厨房、后线下扣款。
5. 追加菜品子订单。
6. 点餐订单列表与详情。

### 第三阶段：预约闭环

1. 用餐预订筛选、餐标详情和价格计算。
2. 住宿预订筛选和价格计算。
3. 散客 15 分钟锁定与支付模拟。
4. 会员提交后占用房间。
5. 预约列表与详情。

### 第四阶段：活动闭环

1. 后台配置模拟数据。
2. 公开或会员专属资格校验。
3. 免费活动报名与自行取消。
4. 收费活动的散客微信支付、会员线下积分扣款。

### 第五阶段：云开发接入

1. 云数据库。
2. 云函数。
3. 微信支付与微信退款。
4. 云打印和厨房通知。
5. 简化后台页面。
