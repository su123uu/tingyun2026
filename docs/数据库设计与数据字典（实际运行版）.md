# 数据库设计与数据字典（实际运行版）

> 基准：`tingyun2026` 小程序、云函数与 `tysj8` 后台。金额单位为人民币元；微信支付时云函数转换为分。

## 1. 核心集合

| 集合 | 用途 |
| --- | --- |
| `users`、`members`、`member_levels` | 微信用户、会员身份和等级 |
| `member_level_benefits`、`member_benefit_accounts`、`member_benefit_usage_records` | 会员权益与住宿次数核销 |
| `meal_items`、`meal_tables`、`meal_table_sessions`、`meal_orders` | 扫码点餐 |
| `dining_rooms`、`dining_standards`、`dining_reservations` | 用餐预订 |
| `accommodation_rooms`、`accommodation_reservations` | 住宿预订 |
| `activity_items`、`activity_signups` | 活动与报名 |
| `notification_subscriptions`、`notification_logs` | 微信订阅消息与企业微信发送记录 |
| `system_settings` | 运行配置 |

## 2. 点餐数据

### 2.1 `meal_table_sessions`

核心字段：`session_id`、`table_id`、`table_name`、`table_area`、`people_count`、`has_order`、`session_status`、`expires_at`、`ordered_at`、`closed_at`、`closed_reason`。

桌台二维码参数为 `t=<table_id>&k=<qr_token>`。存在活动订单的桌台会话会返回 `active_order_no`，扫码后直接打开订单详情。

### 2.2 `meal_orders`

一条记录表示一桌一次用餐；首单和加菜写入 `batches[]`，不会产生独立加菜订单。

| 字段 | 说明 |
| --- | --- |
| `order_no`、`session_id`、`table_*`、`people_count` | 订单和桌台快照 |
| `customer_type`、`member_id`、`customer_name`、`customer_mobile` | 首次提交菜品人的身份快照 |
| `notification_openid` | 点餐状态消息接收人 |
| `batches` | 首单与加菜明细 |
| `total_amount`、`regular_total_amount` | 整桌普通价；前者仅作兼容 |
| `member_total_amount` | 整桌会员价 |
| `checkout_customer_*`、`checkout_openid` | 实际结账人身份快照 |
| `checkout_amount` | 按结账身份确定的实际结账金额 |
| `payment_method`、`payment_trade_no`、`payment_total_fee` | 微信支付或会员积分抵扣/线下核销信息 |
| `points_deduct_amount`、`points_deduct_status`、`member_checkout_confirmed` | 会员点餐确认积分抵扣记录；当前只记录，不自动扣减余额或积分 |
| `order_status` | `preparing`、`completed` |
| `payment_status` | 见第 4 节 |
| `admin_remark`、`user_deleted_at` | 后台备注与用户侧隐藏标记 |

`batches[]`：`batch_no`、`batch_type`、`batch_title`、`items`、`regular_amount`、`member_amount`、`amount`（普通价兼容）、`remark`、`quick_remarks`、`print_status`、`created_at`。

`items[]`：`item_id`、`name`、`image`、`quantity`、`price`/`regular_price`、`member_price`、`amount`/`regular_amount`、`member_amount`。价格均为下单快照，不随后续菜品配置变化。

## 3. 预订、活动与会员

- 用餐和住宿预订保存联系人、会员身份、资源、金额、`reservation_status`、`payment_status` 与支付交易字段。
- 用餐餐标 `dining_standards.dishes` 使用对象结构：`{ "凉菜": ["红油留香鸡", "白玉脆蛰皮"], "热菜": ["蒜爆泉水鱼"] }`；对象 key 为分类名，数组保存具体菜名。
- 住宿订单可保存 `benefit_account_id`、`benefit_discount_amount`；免费住宿权益采用锁定、后台确认核销、取消释放。
- 活动报名保存活动、联系人、人数、身份、单价、金额、`signup_status`、`payment_status`。
- 会员余额和积分仅是线下资料；点餐会员确认积分抵扣时只记录 `points_deduct_amount` 与待核销状态，不自动扣减。

## 4. 统一支付状态

| 值 | 含义 |
| --- | --- |
| `unpaid` | 点餐已出单、尚未进入整桌结账 |
| `pending_wechat_pay` | 预订或活动待发起微信支付 |
| `paying` | 微信支付已发起，等待回调 |
| `offline_pending` | 会员待线下核销；点餐中可与 `order_status=completed` 同时出现，表示已清台但仍待店员核销 |
| `settled` | 微信支付成功或线下核销完成 |
| `cancelled` | 已取消，无需支付 |

点餐不再写入 `pending_payment`、`pending_offline`、`settlement_status` 或批次支付字段。

## 5. 通知与审计

`notification_subscriptions` 保存一次性订阅授权：`openid`、`business_type`、`business_no`、`template_key`、`remaining_count`、`status`、`page`。`notification_logs` 保存微信订阅消息、企业微信应用消息及 Webhook 的发送结果。

`financial_transactions` 与 `operation_logs` 为后续财务和审计扩展集合，当前支付回调以订单支付字段为事实来源。
