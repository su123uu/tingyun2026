# 停云山居小程序 · 设计规范 v2.1

> 更新日期：2026-05-29  
> 适用范围：微信小程序真实开发阶段  
> 当前阶段：前端假数据版，后端、云开发、支付、云打印接口暂未接入

---

## 一、设计定位

停云山居小程序 v1.0 的核心目标是同时完成两件事：一是让到店客人能顺畅完成扫码点餐，二是让新客通过首页、介绍、会员和活动内容感受到山居的气质。

| 维度 | 规范 |
|------|------|
| 风格关键词 | 典雅、安静、留白、东方、克制 |
| 视觉气质 | 山间生活空间，不做过度商业化促销感 |
| 首页节奏 | 参考成熟茶饮小程序的信息效率，但避免直接相似 |
| 页面密度 | 比当前首页更疏朗，模块之间保留呼吸感 |
| 实现优先级 | 直接在微信小程序中实现，不再单独制作完整 HTML 页面 |

首页需要保留高效入口，但不要堆满内容。轮播图、点单/预订按钮、山居介绍、活动入口在视觉重量上保持接近，避免某一块过度抢占首屏。

---

## 二、v1.0 页面结构

底部 Tab 固定为 5 个：

| Tab | 页面 | 角色 |
|-----|------|------|
| 首页 | `pages/home/home` | 品牌展示、快捷入口、会员/活动提示 |
| 点餐 | `pages/menu/menu` | 扫码点餐、购物车、结算占位 |
| 介绍 | `pages/intro/intro` | 山居介绍、空间/住宿/餐厅展示、联系方式 |
| 订单 | `pages/orders/orders` | 假数据订单列表、状态筛选、订单详情入口 |
| 我的 | `pages/profile/profile` | 会员信息、历史消费、权益、客服入口 |

预约预订在 v1.0 作为展示/占位入口存在，可展示住宿、用餐、空间预订类型，但不做真实库存、日历锁房和在线支付。

## 三、色彩系统

| 名称 | 色值 | 用途 |
|------|------|------|
| 山居绿 | `#4F6F52` | 主按钮、点餐主流程、导航选中、品牌强调 |
| 古铜棕 | `#8B7355` | 预约预订模块，用于和点餐区分 |
| 宣纸白 | `#F8F6F1` | 页面底色 |
| 暖米色 | `#E8E4DC` | 卡片底、图片占位、浅色分隔 |
| 装饰金 | `#C9A962` | 会员、细线、徽章、小面积强调 |
| 墨黑 | `#1A1A1A` | 主标题、价格、关键数字 |
| 墨灰 | `#4A4A4A` | 正文、说明文字 |
| 雾灰 | `#7C8A8C` | 辅助说明、空状态、次级图标 |
| 禁用灰 | `#BBBBBB` | 不可点击、未完成状态 |

色彩使用原则：

- 金色只做小面积点缀，避免大面积金色背景。
- 点餐流程以山居绿为主，预约模块以古铜棕为辅。
- 页面背景保持浅暖，不使用冷蓝、亮紫、强渐变。
- 会员卡可以使用深色底，但需要留白和细线控制质感，不做强促销风格。

## 四、字体与排版

微信小程序实际字体以系统字体为主，品牌标题可通过字重、字距和留白体现东方感。

| 用途 | 字号建议 | 字重 | 说明 |
|------|----------|------|------|
| 页面标题 | 40-48rpx | 600 | 首页品牌名、详情页标题 |
| 区块标题 | 30-34rpx | 600 | 点餐分类、活动、会员权益 |
| 卡片标题 | 28-32rpx | 500/600 | 商品名、活动名、房间名 |
| 正文 | 24-28rpx | 400 | 说明、介绍、菜品描述 |
| 辅助文字 | 20-24rpx | 400 | 时间、状态、提示、标签 |
| 价格/数字 | 30-42rpx | 600/700 | 订单金额、会员消费额 |

排版原则：

- 页面左右安全边距建议 `32rpx`。
- 模块间距建议 `32-48rpx`，首屏避免连续卡片堆叠。
- 卡片内部标题和正文间距控制在 `8-16rpx`。
- 不使用 emoji 作为正式图标；可用 TDesign 图标或自定义线性图标。

## 五、首页布局规范

首页目标是“能点单、能预订、能理解山居、能看到活动”，但首屏不追求塞满。

推荐结构：

| 顺序 | 模块 | 规范 |
|------|------|------|
| 1 | 顶部品牌区 | Logo/名称/一句短标语，控制高度，避免大面积空头图 |
| 2 | 轮播图 | 3 张以内，统一比例，主视觉清爽 |
| 3 | 快捷入口 | 点餐、预约两个同尺寸入口，点餐为主色，预约为古铜棕 |
| 4 | 会员卡/权益提示 | 轻量展示等级、折扣、消费进度或权益，不做满屏促销 |
| 5 | 山居介绍 | 一张图 + 短文 + 进入介绍页 |
| 6 | 活动入口 | 2-3 条即可，更多内容进入详情或介绍页 |

尺寸建议：

- 轮播图高度：`300-360rpx`，圆角 `24-32rpx`。
- 两个快捷入口同高：`150-180rpx`。
- 首页卡片圆角：`24-32rpx`。
- 首屏内容最多出现 3 个主要模块：品牌/轮播、快捷入口、会员或活动摘要。

## 六、核心组件规范

### 底部导航

- 5 个 Tab：首页、点餐、介绍、订单、我的。
- 高度约 `112-128rpx`，包含安全区。
- 背景使用半透明白或宣纸白，顶部细分隔线。
- 选中态使用山居绿，会员/金色不要用于 Tab 主选中态。

### 卡片

- 用于商品、订单、活动、会员权益等独立信息块。
- 背景默认白色或半透明白，少量使用暖米色。
- 阴影只做轻微层次：

```css
box-shadow: 0 1px 12px rgba(0, 0, 0, 0.025), 0 0 0 1px rgba(0, 0, 0, 0.03);
```

### 按钮

- 主按钮：山居绿底白字。
- 预约按钮：古铜棕底白字。
- 次级按钮：白底描边或浅米色底。
- 禁用按钮：禁用灰或低透明度。

### 状态标签

订单状态建议：

| 状态 | 颜色 |
|------|------|
| 待支付/待处理 | 装饰金 |
| 已支付/制作中 | 山居绿 |
| 已完成 | 雾灰 |
| 已取消 | 禁用灰 |

## 七、假数据阶段规范

当前没有后端接口，腾讯云开发未购买，所有页面先使用前端假数据实现完整体验。

开发规则：

- 假数据集中放在页面 JS 的 `data` 或后续统一 `mock` 文件中，字段命名按未来接口设计。
- 页面交互必须像真实业务一样完整，例如加购、合计、订单状态、会员等级、历史消费。
- 涉及后端能力的按钮显示真实意图，但给出明确状态反馈。
- 不写死“开发中”作为最终页面主体，除预约这类 v1.0 明确占位模块外。

暂不接入但需要预留的能力：

| 能力 | 当前表现 | 未来接入 |
|------|----------|----------|
| 微信支付 | 前端模拟支付成功/失败 | 云函数 + 微信支付 |
| 云数据库 | 本地假数据/页面状态 | 云开发数据库 |
| 会员注册 | 模拟授权和会员资料 | 微信手机号授权 + 用户表 |
| 厨房打印 | 下单成功展示“已通知厨房” | 云打印机 API |
| 商家通知 | 前端展示通知状态 | 飞书/微信通知 |
| 预约库存 | 仅展示和客服引导 | 房态/桌态/时段管理 |

## 八、业务页设计规则

### 点餐页

- 顶部显示桌号，如果没有扫码参数，则显示“堂食点餐”或模拟桌号。
- 分类保持清晰：套餐、凉菜、热菜、主食、饮品。
- 商品卡片包含名称、描述、价格、加减数量。
- 底部固定购物车栏，显示数量、合计、结算按钮。
- 模拟支付成功后生成假订单，并跳转订单页或订单详情页。

### 介绍页

- 重点展示山居气质和空间信息，不要做成普通企业介绍。
- 内容包括：关于山居、禅修、茶艺、餐饮、住宿、房间/空间列表、联系方式。
- 房间、餐厅、空间资料可使用现有对照表假数据。

### 订单页

- 显示点餐订单为主，预约订单可作为占位或示例。
- 支持状态筛选：全部、待处理、制作中、已完成、已取消。
- 订单卡片包含订单号、时间、桌号、商品摘要、金额、状态。
- 详情页展示商品明细、支付状态、厨房通知/打印状态占位。

### 我的页

- 会员信息是真实字段展示，即使当前是假数据。
- 字段包括：会员等级、折扣、累计消费、历史订单数、活动权益、生日礼遇等。
- 未登录状态展示“成为会员”；模拟登录后展示会员卡和权益。

### 活动详情

- v1.0 不做在线报名表。
- 活动详情展示时间、地点、内容、适合人群。
- 报名按钮使用“电话咨询/联系客服/添加微信”方向。

## 九、验收标准

设计和前端实现完成后，应满足：

- 首页不拥挤，轮播、点餐/预约、山居介绍、活动模块视觉重量均衡。
- 5 个 Tab 都不是空白页，除预约入口外不以“开发中”作为主要内容。
- 点餐页可完成从加购、合计、模拟支付到订单展示的闭环。
- 会员页展示等级、折扣、历史消费、活动权益等真实字段。
- 所有后端未接入能力都有清晰占位，不影响用户理解。
- 视觉统一使用本规范色彩、圆角、间距和状态规则。
