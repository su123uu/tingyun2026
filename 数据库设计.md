# 停云山居小程序数据库设计 v1.0

> 定稿日期：2026-06-03
> 适用阶段：腾讯云开发 / 云数据库接入
> 设计口径：散客线上支付锁定，会员线下积分或补差，所有预约均需人工确认。

## 一、设计结论

1. 正式云数据库采用业务集合，不再只使用前端 mock 阶段的统一本地存储。
2. 预订正式拆成 `dining_reservations` 和 `accommodation_reservations` 两个集合；订单列表和日历由云函数聚合。
3. 散客提交预订后创建 15 分钟临时锁定，字段为 `lock_expires_at`。锁定期间对应房间或包间在用户端置灰，其他人不可选择。
4. 会员提交预订后直接进入待人工确认，并占用房间或包间；后续由店员线下积分扣款，不在用户端展示积分抵扣操作。
5. 用户端不提供预约取消或退款按钮，统一展示联系客服。
6. 活动报名范围由后台配置，可为公开报名或会员专属。
7. 菜品分类不单独建集合，分类字段直接写在 `meal_items` 中。
8. 会员等级、等级权益和会员实际权益账户分开存储；住宿预订页需要读取会员的免费住宿权益。
9. 桌号单独建 `meal_tables` 集合，后台可自由新增、停用、删除桌台，并通过云函数生成可打印二维码。
10. 所有用户端、后台和财务可见的业务编号统一字段名为 `order_no`。
11. 首页轮播图、快捷入口和介绍卡片进入数据库配置，后台可替换图片、文案、排序和上下架。

## 三、通用字段约定

| 字段 | 类型 | 说明 |
|------|------|------|
| `_id` | string | 云数据库文档 ID，可与业务 ID 相同 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |
| `created_by` | string | 创建人用户 ID，系统导入可为空 |
| `is_deleted` | boolean | 软删除标记，默认 `false` |

金额统一使用人民币元，类型为 `number`。时间统一使用 ISO 8601 字符串或云数据库 Date 类型；如果使用 Date 类型，接口层统一格式化为 ISO 字符串给前端。

### 3.1 业务编号规则

用户端、后台和财务对账展示的业务编号统一使用：

```text
业务前缀 + YYYYMMDDHHmmss + 4位随机码
```

| 业务 | 字段 | 前缀 | 示例 |
|------|------|------|------|
| 点餐订单 | `meal_orders.order_no` | `TYMEAL` | `TYMEAL20260603153908A7K2` |
| 用餐预订 | `dining_reservations.order_no` | `TYDINING` | `TYDINING20260603153908B3Q9` |
| 住宿预订 | `accommodation_reservations.order_no` | `TYROOM` | `TYROOM20260603153908C6M1` |
| 活动报名 | `activity_signups.order_no` | `TYACTIVITY` | `TYACTIVITY20260603153908D8P4` |

说明：

- 编号按生成时间带出年份、日期和秒级时间，便于人工识别。
- 末尾 4 位随机码用于降低同秒并发冲突；云端仍需唯一索引兜底。
- 云数据库 `_id` 作为内部主键；用户端、后台和财务统一展示 `order_no`。财务流水另有 `transaction_no`。

### 4.4 `member_level_benefits`

用于配置“某个等级拥有什么权益”。不同等级权益不同、次数不同，都在这里配置。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `level_benefit_id` | string | 是 | 等级权益配置 ID |
| `level_id` | string | 是 | 关联 `member_levels` |
| `benefit_key` | string | 是 | 权益键，如 `free_accommodation` |
| `benefit_name` | string | 是 | 权益名称，如免费住宿 |
| `benefit_type` | enum | 是 | `quota` 次数权益、`service` 服务权益、`discount` 折扣权益 |
| `total_quota` | number | 否 | 总次数或额度，次数权益必填 |
| `quota_unit` | string | 否 | 单位，如 `次`、`人次`、`场` |
| `applies_to` | array | 是 | 适用业务，如 `accommodation`、`activity` |
| `rule` | object | 否 | 规则配置，存放不限房型等扩展规则 |
| `is_enabled` | boolean | 是 | 是否启用 |
| `sort_order` | number | 是 | 排序 |

免费住宿权益配置示例：

```js
{
  level_id: 'level_1',
  benefit_key: 'free_accommodation',
  benefit_name: '免费住宿',
  benefit_type: 'quota',
  total_quota: 10,
  quota_unit: '次',
  applies_to: ['accommodation'],
  rule: {
    room_scope: 'all',
    room_type_limit: [],
    note: '不限房型，仍需检查房间日期可用性并人工确认'
  }
}
```

索引：`level_id + benefit_key`、`benefit_key`、`is_enabled + sort_order`。

| `member_id` | string | 是 | 会员 ID |

### 6.2 `meal_tables`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `table_id` | string | 是 | 桌台 ID，如 `A01` |
| `table_name` | string | 是 | 展示名称，如 `A01 桌`、`露台 1 号桌` |
| `table_area` | string | 否 | 区域，如 `大厅`、`包间`、`露台` |
| `capacity` | number | 否 | 建议容纳人数 |
| `sort_order` | number | 是 | 后台和列表排序 |
| `table_status` | enum | 是 | `enabled`、`disabled`、`deleted` |
| `qr_token` | string | 是 | 二维码校验令牌，生成后写入 |
| `qr_version` | number | 是 | 二维码版本号，重新生成时递增 |
| `qr_scene` | string | 否 | 小程序码 scene 参数 |
| `qr_image_file_id` | string | 否 | 云存储中的二维码图片 fileID |
| `last_qr_generated_at` | datetime | 否 | 最近生成二维码时间 |

二维码建议内容：

```text
TY_TABLE:<table_id>:<qr_token>
```

或使用微信小程序码 `scene`：

```text
t=<table_id>&v=<qr_version>&k=<qr_token>
```

说明：

- 后台新增桌台后，调用云函数生成 `qr_token` 和二维码图片。
- 桌台改名不影响二维码；桌台删除或停用后，扫码应提示“桌码已失效，请联系店员”。
- 桌牌丢失时，不需要删除桌台，只需重新生成二维码：更新 `qr_token`、递增 `qr_version`，旧二维码立即失效。

索引：唯一 `table_id`、唯一 `qr_token`、普通 `table_status + sort_order`、普通 `table_area + sort_order`。

## 十、锁定与占用规则

房间或包间是否可选，由云函数按状态动态判断，不单独依赖前端缓存。

### 10.1 视为占用

| 状态 | 说明 |
|------|------|
| `pending_payment` 且 `lock_expires_at` 未过期 | 散客支付前 15 分钟临时锁定 |
| `paid_pending_confirmation` | 散客已支付，待人工确认 |
| `pending_confirmation` | 会员已提交，待人工确认 |
| `confirmed` | 已确认接待 |

这些状态下，对应房间或包间在用户端置灰，其他用户不可选择。

### 10.2 视为释放

| 状态 | 说明 |
|------|------|
| `pending_payment` 且 `lock_expires_at` 已过期 | 散客超时未支付，自动释放 |
| `rejected` | 无法接待，已拒绝 |
| `refunded` | 散客退款完成 |
| `cancelled` | 客服已取消 |

支付回调、人工确认和后台修改状态时，都必须在事务中再次检查冲突。

## 十五、并发事务要求

### 15.0 桌台二维码生成

1. 后台创建或更新 `meal_tables`。
2. 调用 `diningTableQrGenerate` 生成新的 `qr_token` 和 `qr_version`。
3. 云函数调用微信小程序码能力或生成普通二维码图片，并上传云存储。
4. 写回 `qr_scene`、`qr_image_file_id`、`last_qr_generated_at`。
5. 旧二维码是否继续有效由 `qr_version` 和 `qr_token` 控制；重新生成后旧码失效。

### 15.1 创建用餐预订

1. 校验日期、餐段、人数、餐标和包间容量。
2. 查询 `dining_reservations` 中同日期、同餐段、同包间的占用记录。
3. 排除已释放状态和已过期的临时锁定。
4. 无冲突时创建预订。
5. 散客写入 `pending_payment` 和 `lock_expires_at`；会员写入 `pending_confirmation`。

### 15.2 创建住宿预订

1. 校验入住日期和离店日期。
2. 查询 `accommodation_reservations` 中同房间且日期范围重叠的占用记录。
3. 排除已释放状态和已过期的临时锁定。
4. 如果会员选择使用免费住宿权益，查询 `member_benefit_accounts` 中 `benefit_key = free_accommodation` 且可用次数足够的账户。
5. 免费住宿权益不限房型，但仍必须通过房间日期可用性检查。
6. 无冲突时创建预订。
7. 散客写入 `pending_payment` 和 `lock_expires_at`；会员写入 `pending_confirmation`。
8. 使用免费住宿权益时，在事务中增加 `locked_quota`，减少 `remaining_quota`，并写入 `member_benefit_usage_records` 的 `locked` 记录，关联 `business_order_no = accommodation_reservations.order_no`。

### 15.3 散客支付回调

1. 根据业务单据查询预订。
2. 校验单据仍为 `pending_payment` 且 `lock_expires_at` 未过期。
3. 再次检查房间或包间冲突。
4. 支付成功后改为 `paid_pending_confirmation` 和 `wechat_paid`。
5. 若已过期，拒绝更新并触发退款或原路关闭支付结果。

### 15.4 后台人工确认

1. 查询预订当前状态。
2. 再次检查冲突，避免线下或并发渠道抢占。
3. 无冲突时改为 `confirmed`。
4. 如订单使用会员权益，确认时保持权益锁定，服务完成后改为 `used`；如无法接待，释放已锁定权益。
5. 如无法接待，散客进入 `refunding`，会员进入 `rejected`。
